# Patches

Unified diffs against a pristine `durable-streams-0.1.2`, generated by real
`diff -u` (not hand-authored hunk headers). The whole set applies cleanly and
the patched crate compiles + passes **100 tests** — verified end-to-end by
`scripts/verify-patches.sh` (which extracts a fresh pristine tree, applies
every `patches/*.patch`, and builds). Apply after vendoring:

```bash
scripts/vendor-upstream.sh
cd vendor/durable-streams-0.1.2
for p in ../../patches/*.patch; do patch -p1 < "$p"; done
```

If a patch ever fails (a newer vendored version drifted from 0.1.2), open the
target at the line in `docs/integration-points.md` and re-generate the diff
from a fresh edit rather than forcing fuzz.

## What these add: keyed-by-conversation reads (`?key=`), fast + durable

Server-side keyed reads that isolate one conversation out of a multiplexed
byte-log stream: `Stream-Key: <conv>` on append, `GET /<stream>?key=<conv>`
(with `?offset=` and live modes) to read just that conversation. Correct
(exactly 1/K data, `bench/bench-keyed.sh`), fast (coalesced + resident-cache
serving, ~16k rps vs ~24k full read while returning 50× less data), durable
across restart, and verified against the real `@durable-streams/state`
`createStreamDB` client (`../client-verify/`).

### The patch files

- `keying-handlers.patch` — `handlers.rs`: `?key=` param + `stream-key` header;
  `record_keyed_append` on the durable-append path; `gather_keyed_bytes`
  (coalesce spans → resident-cache-first read → slice in memory) shared by
  catch-up **and** live; `handle_keyed_catchup`; **live keyed reads** for
  long-poll and SSE (offset always advances to the underlying tail so a reader
  skips other keys' data); Linux zero-copy append falls back to buffered for
  keyed appends.
- `keying-store.patch` — `store.rs`: `KeyDir` (in-memory map + durable journal
  under one lock) on `StreamState`; `record_keyed_append`/`keyed_spans`;
  recovery rebuilds from the journal; delete/fork clean up the `.keys` file;
  recover-scan sweeps orphan journals.
- `keying-key_journal-new.patch` — new `key_journal.rs`: the durable per-stream
  `.keys` sidecar (CRC-framed records, torn-tail-safe replay, fsync piggybacked
  on the meta-flush cadence). See its module docs for the exact durability
  guarantee.
- `keying-main.patch` — registers `mod key_journal;`.
- `keying-wal-e2e_tests.patch` — real-HTTP e2e tests: interleaved keyed
  catch-up; keyed index survives a simulated crash/restart; keyed long-poll
  serves filtered backlog and advances past other keys' data; keyed SSE emits
  only the key's frames.

### Honest caveats (unchanged from the design)

- **Durability tail:** a hard crash can lose the un-fsynced journal tail (~100ms
  meta-flush window) — those most-recent keys aren't `?key=`-queryable until
  re-indexed; the append data itself is never at risk (it's in the WAL/log).
- **CPU:** keyed reads use more CPU than a full read (filtering a byte-log means
  reading the coalesced superset and copying out the wanted spans) — the payoff
  is 50× less wire data at near-parity latency.
- **Linux zero-copy guard** is `#[cfg(target_os="linux")]` — hand-reviewed, not
  compiled on macOS; needs a Linux/CI build to confirm.
- **`crates/ds-index`** (Prisma-style fuse-filter segment index) stays a tested
  standalone crate, deliberately NOT wired in — the exact in-memory directory
  makes it unnecessary until the directory outgrows memory or the log spans
  thousands of cold segments (see `../bench/README.md`).
